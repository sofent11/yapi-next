# YApi 全量替换切换与回滚 Runbook

## 1. 目的
- 在“直接全量替换”发布模式下，保证可控切换、快速回滚、数据一致性可验证。

## 2. 角色与职责
- 发布负责人：执行切换、宣告阶段开始/结束。
- DBA：执行 Mongo 快照、恢复演练、索引检查。
- 后端负责人：执行服务切流、健康检查、核心 API 验证。
- 前端负责人：核心页面验证、导入导出任务页验证。
- 观察员：监控 SLI（菜单延迟、导入耗时、导出耗时、错误率）。

## 3. 切换前 24 小时
1. 冻结变更：停止主干高风险改动，仅允许修复性提交。
2. 数据快照：执行 Mongo 全量快照并记录快照 ID。
3. 压测复核：保留最新压测报告（1 万接口菜单、1000 接口导入、OAS3 导出）。
4. 回滚演练：在预发环境完成“恢复快照 + 回切旧服务”演练。
5. 发布包校验：确认构建产物、配置、环境变量一致。

## 4. 切换窗口（建议 15-30 分钟）
1. 进入只读窗口：
- 旧系统关闭写操作（接口编辑/导入/删除/分类管理）。
- 保留只读查询与导出能力。
2. 执行最终增量迁移：
- 运行索引脚本：`npm run db:create-indexes`
- 运行最终校验脚本（接口数量、分类数量、唯一键冲突）。
3. 切流：
- 网关流量从旧服务切到新服务（按环境策略全量或分批）。
4. 切换后立即 Smoke：
- 登录/项目列表/接口菜单
- 导入（small sample）
- 导出（openapi3 + swagger2）
- mock 路由命中

## 5. 切换后 30 分钟观察
- 关键 SLI：
  - `/api/interface/list_menu` P95 < 500ms（1 万接口规模）
  - 1000 接口导入 < 60s
  - 导出 OpenAPI3 P95 < 2s
- 错误监控：
  - 5xx 错误率、Mongo 慢查询、导入失败明细
- 数据一致性：
  - 接口总量、分类归属、`project_id + path + method` 唯一性

## 6. 回滚触发条件
- 任一核心 SLI 连续 10 分钟超阈值。
- 数据一致性校验失败（重复主键冲突、分类错配、接口缺失）。
- 核心链路不可用（导入/导出/mock/menu 任意一条中断）。

## 7. 回滚步骤
1. 立即停止新服务写入并摘除新服务流量。
2. 网关切回旧服务实例。
3. 根据快照 ID 执行 Mongo 恢复。
4. 重放切换窗口写操作日志（若采用双写或事件日志）。
5. 验证旧服务核心链路与数据一致性。

## 8. 切换后复盘
- 记录故障时间线、影响范围、回滚耗时。
- 记录超阈值指标、根因与修复项。
- 输出下一次切换的前置检查清单优化项。
