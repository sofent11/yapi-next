## Design Context

### Users
`app/debugger` serves developers and testers working together around the same local API workspace.
They use it during interface联调, request verification, case沉淀, collection回归, and Git-based collaboration.
The product should help them move from "I need to send one request" to "this flow is now reusable and reviewable" without leaving the tool.

### Brand Personality
Three brand words: mechanical, clear, dependable.

The interface should feel like a calm local instrument panel rather than a marketing-heavy product UI.
It should create confidence, focus, and operational clarity, with quick feedback and very little decorative noise.

### Aesthetic Direction
Direction: industrial workbench in light mode.

This debugger should feel closer to an IDE/control console than a glossy SaaS dashboard.
Visual language should be structured, left-aligned, compact, and intentional, with blue-gray neutrals, restrained accents, and strong information hierarchy.

Anti-references:
- generic GitHub-like white/gray admin styling
- neon dark dashboards
- soft consumer-product cards and playful illustrations
- visually noisy "AI slop" layouts with too many repeated containers

### Design Principles
1. Keep the main workflow visible: import, repair, send, save as case, add to collection, sync.
2. One place for each kind of state: navigation, workspace context, request editing, response actions, Git sync.
3. Make temporary work feel temporary and reusable assets feel durable.
4. Prefer calm density over minimal emptiness or overwhelming control overload.
5. Every empty state or warning must point to the next useful action.
